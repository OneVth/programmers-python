## 문제별 풀이 요약

### #340203 - 심폐소생술

**핵심 아이디어:** 표준 순서 리스트와 입력 리스트를 이중 루프로 매칭 → 일치 시 인덱스+1을 결과에 추가 (1-based 변환)

```python
def solution_v1(cpr: list[str]) -> list[int]:
    answer = []
    basic_order = ["check", "call", "pressure", "respiration", "repeat"]
    for action in cpr:
        for i in range(len(basic_order)):
            if action == basic_order[i]:
                answer.append(i + 1)
    return answer
```

**원본 C++ 빈칸 위치:**
```cpp
for(int i=0; i<___; i++){          // → cpr.size()
    for(int j=0; j<___; j++){      // → basic_order.size()
        if(cpr[i] == basic_order[j]){
            answer[i] = ___;       // → j + 1  (1-based)
            break;
        }
    }
}
```

**함정:** 인덱스가 0-based가 아니라 **1-based** — `j + 1`을 빠뜨리면 `[1, 3, 4, 0, 2]`가 나옴.

### #340204 - 병과분류

**핵심 아이디어:** 문자열의 마지막 4글자를 추출 후 if/elif 체인으로 4개 병과 분기, fallback은 `"direct recommendation"`

```python
def solution_v1(code: str) -> str:
    last_four_words = code[-4:]
    if last_four_words == "_eye":
        return "Ophthalmologyc"
    elif last_four_words == "head":
        return "Neurosurgery"
    elif last_four_words == "infl":
        return "Orthopedics"
    elif last_four_words == "skin":
        return "Dermatology"
    else:
        return "direct recommendation"
```

**원본 C++ 빈칸 위치:** `if/else if` 조건절과 `else` 키워드 자체를 채워 넣는 형태 — 단순 조건문 구조 학습용.

## 오늘의 핵심 패턴

### 1-based vs 0-based 인덱스 변환

문제가 "몇 번째 단계인가"를 묻는 경우 거의 항상 **1-based**.
- Python의 `list.index()`나 `enumerate()`는 0-based 반환 → `+ 1` 필수
- 인덱싱 결과를 그대로 출력/반환하기 전에 변환 시점을 한 번 더 확인하는 습관

```python
# ❌ 0-based 그대로
answer.append(basic_order.index(action))

# ✅ 1-based 변환
answer.append(basic_order.index(action) + 1)
```

### 음수 인덱스 슬라이싱: `s[-4:]`

C++의 `code.substr(code.size()-4, 4)`와 동일한 동작.
- Python 슬라이싱은 **길이가 부족해도 에러 없이** 가능한 범위만 반환 (방어적)
- C++ `substr`은 길이 부족 시 undefined behavior → 그래서 문제에 `4 ≤ code길이` 제약이 명시됨

```python
"abc"[-4:]   # "abc"  (에러 없음)
"abcde"[-4:] # "bcde"
```

### 선형 탐색의 시간복잡도 표기

`basic_order` 길이가 5로 고정인 경우:
- 실질 동작: O(5n) ≈ O(n)
- 일반화 표기: O(n × k) — k가 상수임을 명시할 때

→ Big-O는 실제 성능보다 **알고리즘의 일반적 형태**를 드러내는 표기.

### 데이터 주도 설계 (Data-Driven Design)

if/elif 체인을 dict 매핑으로 바꾸면 **로직과 데이터가 분리**되어 확장에 유리.

```python
# 로직 중심 (병과 추가 시 코드 수정)
if last == "_eye": return "Ophthalmologyc"
elif last == "head": return "Neurosurgery"
...

# 데이터 중심 (병과 추가 시 dict 한 줄만 추가)
DEPT = {
    "_eye": "Ophthalmologyc",
    "head": "Neurosurgery",
    "infl": "Orthopedics",
    "skin": "Dermatology",
}
return DEPT.get(code[-4:], "direct recommendation")
```

→ **개방-폐쇄 원칙(OCP)**: 새 병과 추가 시 코드 수정 없이 데이터만 추가.
